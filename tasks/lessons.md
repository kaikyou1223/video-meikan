# Lessons Learned

## 2026-03-24: デプロイ前に必ずコミットする

**問題**: 複数の機能実装（記事パーツ拡張）を行った後、コミットせずにデプロイした。外部プロセス（linter等）でファイルが巻き戻された際に作業が失われるリスクがあった。

**ルール**:
- 機能実装が一段落したら、デプロイ前に必ずコミットする
- 特にCSS・PHPの変更は外部ツールで上書きされるリスクがあるため、こまめにコミット
- デプロイスクリプトの先頭で未コミット変更がないかチェックする仕組みを検討

**対策**:
- ルートに `.gitignore` を追加（機密ファイル・ビルド成果物を除外）
- `secret.md`, `*.zip` 等をトラッキング対象外に

## 2026-03-24: バッチ実行順序の厳守（作品画像が女優画像に混入）

**問題**: `fetch_fanza.php` が `actresses.thumbnail_url` を作品のパッケージ画像から設定していた。女優ポートレート画像を取得する仕組みがなく、全女優のサムネイルがDVDジャケットになっていた。

**ルール**:
- バッチは必ず `run_all.php` 経由で実行する（登録→プロフィール→作品の順序を保証）
- `fetch_fanza.php` に女優サムネイル設定ロジックを入れない（プロフィールバッチの責務）
- 個別バッチを単体実行する提案をしない

**対策**:
- `fetch_actress_profiles.php` を新規作成（FANZA ActressSearch API で女優画像を取得）
- `run_all.php` を新規作成（実行順序を保証）
- `fetch_fanza.php` から女優サムネイル設定ロジックを削除

## 2026-03-24: CSS変更は本番確認を必ず行う

**問題**: コミット 51ad22f で genre-card の padding/arrow位置、work-card の画像サイズ/CTAスタイルを変更したが、本番で矢印がジャンルテキストと被り、ジャンルページの表示が崩れた。

**ルール**:
- CSSの変更は必ず本番環境で目視確認する
- 複数コンポーネントのCSS変更を1コミットにまとめない（問題の切り分けが困難になる）
- 既存ページに影響するCSS変更は、変更前後のスクリーンショットで比較する
